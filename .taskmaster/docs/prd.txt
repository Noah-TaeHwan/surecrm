# Overview  
**SureCRM**은 보험설계사를 위한 AI 기반 관계 중심 고객관리 솔루션으로, 기존 CRM의 한계를 뛰어넘어 보험업계의 핵심 영업 방식인 '소개 네트워크'를 시각화하고 관리할 뿐만 아니라, 첨단 AI 리드 제너레이션 엔진을 통해 실시간으로 잠재 고객을 발굴하고 최적의 영업 전략을 제시하는 전문화된 플랫폼입니다.

## 해결하는 문제
1. **기존 네트워크 한계**: 보험설계사들의 영업 성공은 고객 간 소개 관계에 크게 의존하지만, 기존 지인 네트워크가 한계에 도달했을 때 새로운 잠재 고객을 발굴하는 것이 극도로 어려움
2. **소개 관계 관리 부재**: 현재 이를 체계적으로 관리할 수 있는 도구가 부족하여 대부분 메모앱, 엑셀, 수기 노트 등 파편화된 방식으로 관리
3. **통합 업무 환경 부재**: 리드 발굴, 관계 관리, 영업 프로세스가 각각 분리되어 있어 종합적인 영업 전략 수립 불가

## 타겟 사용자
- **1차 타겟**: 1-3년차 개인 보험설계사 (디지털 도구에 익숙하지만 기존 CRM이 보험업 특성에 맞지 않다고 느끼는 사용자)
- **2차 타겟**: 중견 보험설계사 및 팀장급 (팀 단위 협업 및 관리 기능 활용)
- **초기 검증 그룹**: AZ보험사 1년차 설계사와 그의 팀원들 (9-10명 규모)

## 핵심 가치 제안
1. **AI 기반 리드 제너레이션**: 실시간으로 라이프 이벤트를 감지하여 보험 니즈가 높은 잠재 고객을 자동 발굴
2. **소개 네트워크 시각화**: 옵시디언 스타일의 그래프 뷰로 고객 간 소개 관계를 직관적으로 시각화
3. **통합 업무 환경**: 리드 발굴부터 계약 완료까지 전체 영업 프로세스를 하나의 플랫폼에서 관리
4. **실시간 AI 코칭**: 상담 과정에서 실시간으로 최적의 접근 전략과 대화 스크립트 제공
5. **초대 기반 성장**: 클럽하우스 스타일의 초대장 시스템으로 양질의 사용자 커뮤니티 구축

# Core Features  

## 1. AI 리드 제너레이션 엔진 (핵심 차별화 기능)

### 실시간 라이프 이벤트 감지 시스템
- **기능**: AI가 공개된 데이터를 분석하여 보험 니즈가 발생하는 라이프 이벤트를 실시간으로 감지
- **감지 이벤트**: 부동산 거래(신혼집 계약, 주택 구매, 이사), 가족 이벤트(결혼, 출산, 자녀 입학), 직업 변화(이직, 승진, 창업), 자산 변화(차량 구매, 고액 자산 취득)
- **데이터 소스**: 부동산 거래 공고, 출생신고 공시, 법인 등기, 차량 등록 등 합법적 공개 정보
- **중요한 이유**: 보험 니즈는 특정 라이프 이벤트와 직결되며, 이벤트 발생 후 7-14일 내가 계약 골든타임

### AI 리드 스코어링 및 우선순위 시스템
- **기능**: 발굴된 잠재 고객에 대한 종합적 분석을 통해 성공 확률과 우선순위 자동 산정
- **스코어링 요소**: 라이프 이벤트 유형 및 시급성(40%), 추정 소득 수준 및 보험료 지불 능력(25%), 기존 보험 가입 현황 및 보장 공백(20%), 접촉 가능성 및 커뮤니케이션 선호도(15%)
- **우선순위 분류**: HOT(90-100점), WARM(70-89점), COLD(50-69점)
- **중요한 이유**: 한정된 시간을 가장 성공 확률이 높은 리드에 집중할 수 있도록 지원

### 개인화 접근 전략 엔진
- **기능**: 각 잠재 고객의 성향과 상황에 맞춤화된 접근 전략 자동 생성
- **성향 분석**: 연령, 직업, 지역, 라이프스타일 기반 커뮤니케이션 스타일 분석
- **최적 접촉법**: 채널(카톡/전화/이메일), 시간대, 메시지 톤앤매너 추천
- **대화 시나리오**: 첫 인사부터 계약 클로징까지 단계별 대화 스크립트 제공
- **중요한 이유**: 개인화된 접근으로 첫 상담 성공률을 극대화

### 예측 분석 및 성공률 시스템
- **기능**: 머신러닝 기반으로 각 리드의 계약 성공 확률과 예상 수수료 예측
- **성공률 예측**: 과거 유사 고객 데이터 기반 계약 확률 산출(오차율 6% 이내 목표)
- **수수료 예측**: 보험 유형별, 고객 프로필별 예상 수수료 금액
- **타임라인 예측**: 첫 접촉부터 계약 완료까지 예상 소요 기간
- **중요한 이유**: 데이터 기반 의사결정으로 영업 효율성 극대화

### 자동화 커뮤니케이션 시스템
- **기능**: AI가 생성한 개인화 메시지를 최적 타이밍에 자동 발송
- **메시지 자동 생성**: 고객별 맞춤형 첫 연락 메시지, 팔로업 메시지 생성
- **발송 타이밍 최적화**: 고객의 라이프패턴 분석을 통한 최적 발송 시간 선택
- **반응 모니터링**: 읽음 여부, 반응 시간, 긍정/부정 반응 실시간 분석
- **중요한 이유**: 24시간 자동화된 리드 너처링으로 기회 손실 최소화

## 2. 소개 네트워크 시각화

### 그래프 뷰 시스템
- **기능**: 옵시디언 스타일의 인터랙티브 네트워크 그래프로 고객 간 소개 관계 시각화
- **노드 표현**: 각 고객을 노드로 표시, 크기는 소개 건수에 비례
- **엣지 표현**: 소개 관계를 화살표로 연결, 방향성 표시
- **색상 코딩**: 영업 단계별, 중요도별, 보험 유형별 색상 구분
- **인터랙티브 조작**: 줌인/줌아웃, 드래그, 노드 클릭으로 상세 정보 표시
- **중요한 이유**: 복잡한 소개 관계를 시각적으로 파악하여 핵심 소개자 식별, 잠재 영업 기회 발견 가능

### 네트워크 분석 기능
- **기능**: 소개 네트워크의 정량적 분석을 통한 인사이트 제공
- **중심성 분석**: 네트워크에서 가장 영향력 있는 노드 식별
- **클러스터 분석**: 연관성 높은 고객 그룹 자동 탐지
- **경로 분석**: A→B→C 형태의 간접 소개 경로 추적
- **성장 패턴**: 시간에 따른 네트워크 확장 패턴 분석
- **중요한 이유**: 데이터 기반으로 네트워크 확장 전략 수립 가능

### 필터링 및 검색
- **기능**: 복잡한 네트워크에서 원하는 정보에 빠른 접근
- **단계별 필터**: 영업 단계별 노드 표시/숨기기
- **깊이 필터**: 1촌/2촌/전체 소개 관계 범위 설정
- **검색 기능**: 고객명 검색 시 해당 노드 중앙 배치 및 하이라이트
- **시간 필터**: 특정 기간 내 소개 관계만 표시
- **중요한 이유**: 대용량 네트워크에서도 효율적인 정보 탐색 가능

## 3. 고객 관리 시스템

### 고객 프로필 관리
- **기능**: 보험설계사 업무에 특화된 고객 정보 관리
- **기본 정보**: 이름, 연락처, 주소, 직업, 소득 수준
- **보험 특화 정보**: 키, 몸무게, 운전 여부, 건강 상태
- **태그 시스템**: 고객 분류 및 검색을 위한 유연한 태그 관리
- **중요도 설정**: High/Medium/Low 3단계 중요도 분류
- **중요한 이유**: 보험 상품 특성상 고객의 신체 정보, 직업, 소득 수준 등이 상품 추천과 직결

### 보험 유형별 특화 관리
- **기능**: 보험 상품별 특수 요구사항 관리
- **태아보험**: 출생예정일, 임신과정, 유산방지제 복용 여부, 태아 이상소견, 장애검사 이상소견
- **자동차보험**: 차량번호, 차량 소유자 정보, 필수 서류 체크리스트
- **생명보험/건강보험**: 기존 보험 가입 이력, 건강 상태, 가족력
- **중요한 이유**: 각 보험 유형마다 필요한 정보와 서류가 다르므로 동적 관리 필요

### 문서 및 파일 관리
- **기능**: 고객별 필수 서류 관리 시스템
- **파일 업로드**: 드래그앤드롭으로 간편 업로드
- **서류 분류**: 신분증, 차량등록증, 보험증권 등 자동 분류
- **체크리스트**: 보험 유형별 필수 서류 완성도 표시
- **안전한 저장**: Supabase Storage 활용한 암호화 저장
- **중요한 이유**: 보험 계약 시 다양한 서류가 필요하므로 체계적 관리 필수

## 4. 영업 파이프라인 관리

### AI 강화 칸반보드 시스템
- **기능**: AI 인사이트가 통합된 시각적 파이프라인
- **기본 단계**: 첫 상담 → 니즈 분석 → 상품 설명 → 계약 검토 → 계약 완료
- **스마트 단계 추천**: AI가 고객 상황 분석하여 최적 다음 단계 추천
- **자동 체류 시간 알림**: 특정 단계에 너무 오래 머무는 고객 자동 감지
- **성공률 기반 정렬**: 각 단계 내에서 성공 확률 순으로 고객 자동 정렬
- **예상 매출 표시**: 각 고객별 예상 수수료를 카드에 실시간 표시
- **중요한 이유**: 체계적인 영업 프로세스 관리로 전환율 향상

### AI 상담 어시스턴트
- **기능**: 상담 과정에서 실시간 AI 지원
- **실시간 대화 분석**: 통화 내용을 실시간으로 분석하여 고객 반응 파악
- **다음 멘트 추천**: 고객 반응에 따른 최적 다음 멘트 실시간 제안
- **객션 핸들링**: 고객 반대 의견에 대한 효과적 대응 방법 즉시 제공
- **클로징 타이밍**: 계약 제안 최적 타이밍 AI 알림
- **중요한 이유**: 실시간 코칭으로 상담 성공률 극대화

### 성과 분석
- **기능**: 파이프라인 효율성 분석
- **전환율 분석**: 단계별 고객 전환율 추적
- **체류 시간**: 각 단계별 평균 체류 시간 분석
- **병목 지점**: 진행이 정체되는 단계 식별
- **목표 대비 성과**: 월/분기 목표 대비 달성률 시각화
- **중요한 이유**: 데이터 기반으로 영업 프로세스 개선점 파악

## 5. 구글 캘린더 연동

### 양방향 동기화
- **기능**: 구글 캘린더와 SureCRM 간 완전 동기화
- **OAuth 2.0 인증**: 안전한 구글 계정 연결
- **실시간 웹훅**: 구글 캘린더 변경사항 즉시 반영
- **충돌 해결**: 동시 수정 시 충돌 감지 및 해결 UI
- **배치 동기화**: 대량 이벤트 효율적 처리
- **중요한 이유**: 기존 일정 관리 도구와의 완벽한 통합으로 업무 연속성 보장

### 미팅 관리 시스템
- **기능**: 고객 미팅 특화 일정 관리
- **미팅 유형 분류**: 첫 상담, 상품 설명, 계약 체결, 후속 관리
- **고객 연동**: 미팅과 고객 정보 자동 연결
- **준비 체크리스트**: 미팅 유형별 사전 준비사항 자동 생성
- **미팅 노트**: 상담 내용, 고객 반응, 다음 액션 기록
- **중요한 이유**: 체계적인 미팅 관리로 상담 품질 향상

### 알림 및 리마인더
- **기능**: 놓치기 쉬운 일정 관리
- **스마트 알림**: 미팅 30분 전, 하루 전 자동 알림
- **준비 알림**: 필요 서류, 상품 자료 준비 리마인더
- **팔로업 알림**: 미팅 후 정해진 시간에 후속 조치 알림
- **모바일 푸시**: 앱을 통한 실시간 알림
- **중요한 이유**: 중요한 일정을 놓치지 않도록 보장

## 6. 팀 관리 및 협업

### 팀 구성 관리
- **기능**: 보험설계사 팀 조직 관리
- **역할 기반 권한**: 팀장, 시니어, 주니어별 차등 권한
- **데이터 공유 설정**: 팀 내 고객 데이터 공유 범위 설정
- **멘토링 시스템**: 선배-후배 설계사 간 지식 공유 지원
- **팀 성과 대시보드**: 팀 전체 영업 성과 통합 관리
- **중요한 이유**: 체계적인 팀 관리로 조직 전체 성과 향상

### 협업 도구
- **기능**: 팀원 간 효율적 협업 지원
- **공유 파이프라인**: 팀 공통 영업 프로세스 표준화
- **고객 이관**: 담당자 변경 시 완전한 데이터 이관

# Technical Architecture  

## 시스템 아키텍처 개요

### 프론트엔드 (Frontend)
- **프레임워크**: React Router v7 + TypeScript
- **상태 관리**: React Query (서버 상태) + Zustand (클라이언트 상태)
- **UI 컴포넌트**: Shadcn/ui + Tailwind CSS
- **차트 라이브러리**: Recharts (성과 대시보드), D3.js (네트워크 그래프 시각화)
- **빌드 도구**: Vite
- **모바일**: PWA 지원으로 네이티브 앱 수준의 모바일 경험 제공

### 백엔드 (Backend)
- **인프라**: Supabase (PostgreSQL + Authentication + Storage + Edge Functions)
- **API**: RESTful API + Real-time subscriptions
- **인증**: Supabase Auth + Row Level Security (RLS)
- **파일 저장**: Supabase Storage (문서, 이미지, AI 분석 결과)
- **실시간 기능**: Supabase Realtime (리드 알림, 팀 협업)

### AI/ML 컴포넌트
- **데이터 수집 엔진**: 
  - 웹 스크래핑 봇 (합법적 공개 정보만 수집)
  - 정부 공시 API 연동 (부동산거래관리시스템, 법인등기시스템)
  - Context7 MCP 활용: 최신 데이터 소스 및 API 업데이트 정보 실시간 반영
- **이벤트 감지 AI**: 
  - 자연어 처리 (NLP) 엔진: 공고문에서 라이프 이벤트 추출
  - 패턴 인식 알고리즘: 유사한 이벤트 패턴 자동 감지
  - 실시간 스트리밍 데이터 처리: Apache Kafka + Apache Storm
- **예측 모델**: 
  - 성공률 예측 (Random Forest + Neural Network 앙상블)
  - 고객 생애가치 예측 (LTV Prediction using XGBoost)
  - 최적 접촉 타이밍 예측 (Time Series Analysis using LSTM)

### 외부 연동 (External Integrations)
- **구글 캘린더**: OAuth 2.0 + Google Calendar API
- **Context7 MCP**: 실시간 데이터 소스 업데이트 및 시장 인텔리전스
- **이메일**: 초대장 발송, 자동 메시지 (SMTP)
- **SMS/카카오톡**: 긴급 알림 및 자동 메시지 발송
- **푸시 알림**: 웹 푸시 + 모바일 푸시 (PWA)

### 주요 관계 및 제약조건
- **User ↔ Client**: 1:N (한 설계사가 여러 고객 관리)
- **Client ↔ Client**: N:N (고객 간 소개 관계, self-referencing)
- **Client ↔ PipelineStage**: N:1 (고객의 현재 영업 단계)
- **User ↔ Team**: N:1 (팀 소속 관계)
- **User ↔ Invitation**: 1:N (초대장 발급 관계)
- **AI_Lead ↔ Client**: 1:1 (AI 발굴 리드가 고객으로 전환)
- **AI_Event ↔ AI_Lead**: 1:N (하나의 이벤트에서 여러 리드 발생 가능)

### 데이터 보안 및 개인정보 보호
- **RLS 정책**: 모든 테이블에 Row Level Security 적용
  - 개인 데이터: 본인만 접근 가능
  - 팀 데이터: 팀 구성원만 접근 가능 (권한 레벨별 차등)
- **역할 기반 접근**: agent(일반 설계사), team_admin(팀장), system_admin(시스템 관리자)
- **데이터 암호화**: 주민등록번호, 계좌정보 등 민감 정보 pgsodium 암호화
- **감사 로그**: 중요 데이터 접근/변경 기록 (GDPR 준수)
- **데이터 보존 정책**: 비활성 사용자 데이터 자동 삭제 (2년 후)

## 인프라 요구사항

### 성능 요구사항
- **API 응답 시간**: REST API 평균 200ms 이하, AI 분석 API 500ms 이하
- **AI 실시간 처리**: 라이프 이벤트 감지 5분 이내, 리드 스코어링 10초 이내
- **동시 사용자**: 1,000명 동시 접속 지원
- **데이터 처리**: 10만 고객 데이터 실시간 처리
- **네트워크 그래프**: 1,000개 노드까지 부드러운 렌더링 (60fps)
- **파일 업로드**: 10MB 파일 업로드 30초 이내 완료

### 확장성 고려사항
- **데이터베이스**: PostgreSQL 파티셔닝으로 대용량 데이터 처리
  - 고객 데이터: 연도별 파티셔닝
  - AI 로그 데이터: 월별 파티셔닝
- **캐싱**: Redis 활용한 자주 조회되는 데이터 캐싱
  - 리드 스코어: 1시간 캐시
  - 네트워크 그래프 데이터: 30분 캐시
- **CDN**: 정적 파일 및 이미지 전송 최적화
- **로드 밸런싱**: 트래픽 증가 시 서버 수평 확장
- **마이크로서비스**: AI 엔진을 별도 서비스로 분리 (향후 계획)

### 보안 요구사항
- **네트워크 보안**: HTTPS 강제, TLS 1.3 이상
- **데이터 암호화**: 
  - 전송 중: TLS 암호화
  - 저장 시: AES-256 암호화 (민감 정보)
- **접근 제어**: 
  - API 레이트 리미팅 (사용자별 분당 100회)
  - IP 화이트리스트 (관리자 기능)
- **모니터링**: 
  - 실시간 시스템 상태 모니터링 (Grafana + Prometheus)
  - 보안 이벤트 로깅 및 알림
- **백업**: 
  - 데이터베이스: 실시간 복제 + 일일 스냅샷
  - 파일: 클라우드 스토리지 자동 백업
  - 재해복구: RTO 4시간, RPO 1시간

### 모니터링 및 알림 시스템
- **성능 모니터링**: API 응답시간, 에러율, 처리량
- **AI 모델 모니터링**: 예측 정확도, 모델 드리프트 감지
- **비즈니스 메트릭**: 리드 전환율, 사용자 활성도, 수익 지표
- **인프라 모니터링**: CPU, 메모리, 디스크, 네트워크 사용률
- **알림 시스템**: 
  - 긴급: 즉시 SMS/이메일 발송
  - 경고: 슬랙 채널 알림
  - 정보: 대시보드 표시

# Logical Dependency Chain

## 개발 순서의 논리적 근거

### 1단계: AI 리드 발굴 시스템 (최우선 개발)
**논리적 근거**: 제품의 핵심 차별화 요소이며, 사용자가 즉시 가치를 느낄 수 있는 유일한 기능

#### 세부 개발 순서
1. **Context7 MCP 통합** (1주차)
   - 이유: 모든 실시간 데이터의 기반이 되는 인프라
   - 결과: 안정적인 데이터 수집 파이프라인 확보

2. **기본 라이프 이벤트 감지** (2주차)
   - 이유: 리드 발굴의 핵심 엔진
   - 결과: 부동산 거래, 출산, 차량 구매 이벤트 감지

3. **리드 스코어링 모델** (3주차)
   - 이유: 발굴된 리드의 우선순위 결정
   - 결과: HOT/WARM/COLD 분류 시스템

4. **기본 사용자 시스템** (4주차)
   - 이유: 리드 정보를 사용자에게 전달하기 위한 기반
   - 결과: 로그인, 회원가입, 초대장 시스템

5. **AI 대시보드 MVP** (5-6주차)
   - 이유: 사용자가 AI 가치를 경험할 수 있는 인터페이스
   - 결과: 리드 목록, 스코어 표시, 기본 필터링

**왜 이 순서인가?**
- 사용자는 1단계 완료만으로도 "새로운 리드가 자동으로 발굴되는" 핵심 가치를 경험
- 기존 CRM이나 네트워크 시각화 없이도 독립적으로 가치 제공 가능
- 투자 대비 사용자 만족도가 가장 높은 기능부터 구현

### 2단계: 기본 CRM 기능 (AI 리드 활용 기반)
**논리적 근거**: AI가 발굴한 리드를 효과적으로 관리하기 위한 필수 도구

#### 세부 개발 순서
1. **고객 데이터 모델** (1주차)
   - 이유: 모든 고객 관리 기능의 기반
   - 의존성: AI 리드 데이터와 호환되는 구조 필요

2. **고객 CRUD 기능** (2주차)
   - 이유: AI 리드를 고객으로 전환하는 기본 기능
   - 의존성: 1단계에서 발굴된 리드 정보 활용

3. **기본 파이프라인** (3주차)
   - 이유: 리드에서 계약까지의 진행 상황 관리
   - 의존성: 고객 정보가 먼저 존재해야 함

4. **데이터 임포트** (4주차)
   - 이유: 기존 고객 데이터와 AI 리드의 통합
   - 의존성: 고객 관리 시스템이 안정화된 후

**왜 이 시점인가?**
- AI 리드만으로는 완전한 고객 관리 불가능
- 사용자들은 기존 고객 데이터도 함께 관리하고 싶어함
- AI 리드 → 고객 전환 프로세스가 필요함

### 3단계: AI 고도화 (정확도 및 개인화)
**논리적 근거**: 기본 AI 시스템의 한계를 극복하고 실용성 극대화

#### 세부 개발 순서
1. **고급 스코어링 모델** (1주차)
   - 이유: 기본 규칙 기반에서 ML 기반으로 업그레이드
   - 의존성: 1단계에서 수집된 리드 데이터 활용

2. **개인화 전략 엔진** (2-3주차)
   - 이유: 단순 리드 발굴에서 실제 상담 성공으로 진화
   - 의존성: 2단계에서 관리된 고객 정보 활용

3. **예측 분석 시스템** (3-4주차)
   - 이유: 데이터 기반 의사결정 지원
   - 의존성: 충분한 고객 데이터와 AI 리드 성과 데이터 필요

4. **실시간 알림 시스템** (4-5주차)
   - 이유: 골든타임 놓치지 않기 위한 즉시성
   - 의존성: 고도화된 AI 분석 결과 기반

**왜 이 시점인가?**
- 1-2단계를 통해 충분한 데이터가 축적됨
- 사용자들이 기본 기능에 익숙해져 고급 기능에 대한 니즈 발생
- AI의 정확도 향상이 사용자 만족도에 직결되는 시점

### 4단계: 소개 네트워크 시각화 (기존 자산 활용)
**논리적 근거**: AI 신규 리드와 기존 네트워크의 시너지 창출

#### 세부 개발 순서
1. **소개 관계 데이터 모델** (1주차)
   - 이유: 복잡한 관계 데이터의 효율적 저장
   - 의존성: 2단계의 고객 데이터 구조 활용

2. **그래프 시각화 엔진** (2주차)
   - 이유: 복잡한 관계를 직관적으로 표현
   - 의존성: 충분한 고객 관계 데이터 필요

3. **네트워크 분석 기능** (3주차)
   - 이유: 시각화를 넘어선 실용적 인사이트 제공
   - 의존성: 그래프 데이터 구조 완성 후

4. **AI 리드와 네트워크 통합** (4주차)
   - 이유: 신규 리드와 기존 네트워크의 연결점 발견
   - 의존성: 3단계의 AI 분석과 네트워크 분석 결합

**왜 이 시점인가?**
- AI 리드 발굴이 안정화된 후 기존 네트워크와의 연결 모색
- 사용자들이 신규 리드에 만족한 후 기존 관계 관리에 관심
- 충분한 고객 데이터가 축적되어야 의미있는 네트워크 구성 가능

### 5단계: 구글 캘린더 연동 (업무 통합)
**논리적 근거**: 외부 도구 의존성을 최소화한 후 안전한 통합

#### 세부 개발 순서
1. **OAuth 인증 시스템** (1주차)
   - 이유: 안전한 외부 연동의 기반
   - 의존성: 기본 사용자 시스템 안정화 필요

2. **양방향 동기화** (2주차)
   - 이유: 캘린더 연동의 핵심 기능
   - 의존성: 내부 미팅 관리 기능 선행 필요

3. **미팅 관리 시스템** (3주차)
   - 이유: 단순 일정 동기화를 넘어선 미팅 특화 기능
   - 의존성: 고객 정보와 AI 분석 결과 연동

4. **알림 및 자동화** (4주차)
   - 이유: 업무 효율성 극대화
   - 의존성: 모든 기본 기능의 안정화 필요

**왜 이 시점인가?**
- 내부 기능이 충분히 안정화된 후 외부 연동
- 사용자들이 플랫폼에 익숙해진 후 편의 기능 추가
- 외부 API 의존성으로 인한 리스크를 최소화

### 6단계: 팀 관리 (개인→팀 확장)
**논리적 근거**: 개인 사용자의 성공 사례 확보 후 조직 확장

#### 세부 개발 순서
1. **팀 구성 관리** (1-2주차)
   - 이유: 팀 기능의 기반 인프라
   - 의존성: 개인 기능의 완전한 안정화 필요

2. **권한 및 데이터 공유** (2-3주차)
   - 이유: 팀 협업의 핵심 요소
   - 의존성: 모든 개인 기능이 팀 환경에서 작동해야 함

3. **팀 성과 대시보드** (3-4주차)
   - 이유: 팀 가치를 증명하는 핵심 기능
   - 의존성: 개인별 성과 데이터 충분히 축적 필요

4. **협업 도구** (4-5주차)
   - 이유: 팀워크 극대화를 위한 부가 기능
   - 의존성: 기본 팀 관리 기능 완성 후

**왜 이 시점인가?**
- 개인 사용자의 성공 사례가 확보되어야 팀 확산 가능
- 모든 개인 기능이 팀 환경에서도 안정적으로 작동해야 함
- 팀 기능의 복잡성으로 인한 버그가 개인 기능에 영향 주지 않음

### 7단계: 완전 자동화 (AI 완전체)
**논리적 근거**: 모든 기능이 안정화된 후 최고 수준의 AI 경험 제공

**왜 마지막인가?**
- 가장 복잡하고 고도화된 AI 기능
- 이전 단계들의 데이터와 사용자 피드백이 필수
- 실패해도 핵심 가치에는 영향 없는 부가 기능

## 각 단계별 사용 가능성 및 독립성

### 단계별 독립적 가치 제공
- **1단계 완료**: "AI가 자동으로 리드를 찾아준다" (혁신적 가치)
- **2단계 완료**: "AI 리드를 체계적으로 관리할 수 있다" (실용적 가치)  
- **3단계 완료**: "AI가 상담 전략까지 제시해준다" (전략적 가치)
- **4단계 완료**: "기존 네트워크와 신규 리드를 통합 관리" (종합적 가치)
- **5단계 완료**: "모든 업무 도구가 하나로 통합" (편의성 가치)
- **6단계 완료**: "팀 전체 성과 향상" (조직적 가치)
- **7단계 완료**: "거의 자동화된 영업 환경" (미래적 가치)

### 점진적 가치 증가 전략
각 단계마다 사용자가 명확히 느낄 수 있는 가치 증분을 제공하여:
- 개발 과정에서 지속적인 사용자 피드백 확보
- 조기 수익화 및 투자 유치 가능성 향상
- 경쟁사 대비 지속적인 우위 유지
- 사용자 이탈률 최소화

# Risks and Mitigations  

## 기술적 리스크 (Technical Risks)

### 리스크 1: AI 예측 정확도 문제
**상황**: 리드 스코어링이나 성공률 예측이 부정확하여 사용자 신뢰도 하락 및 실제 영업 성과 저하
**영향도**: 매우 높음 (핵심 기능의 신뢰성 문제로 제품 전체 실패 가능성)
**발생 확률**: 높음 (AI 모델 초기 단계에서는 불가피)
**예상 손실**: 사용자 이탈률 70% 이상, 매출 90% 감소

**완화 방안**:
- **점진적 정확도 개선 전략**: 초기에는 보수적 예측(70% 정확도)으로 시작하여 데이터 축적에 따라 점진적 정교화 (목표: 6개월 내 85% 정확도)
- **투명한 신뢰도 표시**: 모든 예측 결과와 함께 신뢰도 구간 명시 ("성공률 75% ± 15%")
- **지속적 학습 시스템**: 실제 결과 피드백을 통한 모델 주간 단위 업데이트
- **사용자 교육 및 기대치 관리**: AI는 보조 도구임을 명확히 하고 최종 판단은 사용자 몫임을 지속적으로 강조
- **대안 시나리오**: 정확도 70% 미만 시 규칙 기반 시스템으로 일시 전환

### 리스크 2: Context7 MCP 의존성 리스크
**상황**: Context7 MCP 서비스 중단, API 변경, 또는 비용 급증으로 인한 핵심 데이터 수집 기능 마비
**영향도**: 높음 (실시간 리드 발굴 기능 중단)
**발생 확률**: 중간 (외부 서비스 의존성)
**예상 손실**: 신규 리드 발굴 100% 중단, 사용자 만족도 급락

**완화 방안**:
- **다중 데이터 소스 확보**: Context7 MCP 외 최소 3개 이상의 백업 데이터 소스 구축
  - 정부 공시 API 직접 연동
  - 부동산 중개 업체 파트너십
  - 웹 스크래핑 자체 개발
- **로컬 캐싱 시스템**: 중요 정보의 7일간 로컬 캐싱으로 단기 서비스 중단 대비
- **그레이스풀 디그레이데이션**: MCP 중단 시에도 기존 고객 관리 기능은 정상 작동 보장
- **SLA 관리**: Context7과 99.5% 가용성 서비스 수준 협약 체결 및 보상 조항 포함
- **자체 개발 로드맵**: 12개월 내 핵심 데이터 수집 기능 자체 개발 계획

### 리스크 3: 네트워크 그래프 성능 문제
**상황**: 고객 수 증가(5,000명 이상)에 따른 네트워크 그래프 렌더링 성능 저하로 사용성 급격히 악화
**영향도**: 중간 (핵심 차별화 기능 사용성 저하)
**발생 확률**: 높음 (대용량 데이터 처리 시 일반적 문제)
**예상 손실**: 그래프 기능 사용률 50% 감소, 차별화 요소 약화

**완화 방안**:
- **단계적 로딩 및 가상화**: 화면에 보이는 노드만 렌더링, 스크롤 시 동적 로딩
- **AI 기반 스마트 필터링**: AI가 사용자 의도를 파악하여 관련성 높은 노드만 우선 표시
- **클러스터링 및 추상화**: 유사한 노드들을 그룹화하여 표시, 필요시 확장
- **성능 모니터링 시스템**: 렌더링 시간 실시간 추적, 5초 초과 시 자동 최적화 모드 전환
- **대안 뷰 제공**: 그래프 외에 테이블, 트리 뷰 등 대안적 시각화 방법 제공

### 리스크 4: 데이터베이스 성능 및 확장성 한계
**상황**: 사용자 수 증가(1,000명 이상)와 데이터 축적으로 인한 데이터베이스 성능 저하
**영향도**: 높음 (전체 서비스 영향)
**발생 확률**: 중간
**예상 손실**: 서비스 응답속도 10배 저하, 사용자 이탈률 40% 증가

**완화 방안**:
- **데이터베이스 최적화 전략**:
  - 자주 조회되는 데이터에 대한 인덱스 최적화
  - 파티셔닝을 통한 대용량 테이블 분할 (고객 데이터: 연도별, AI 로그: 월별)
  - 쿼리 성능 지속 모니터링 및 슬로우 쿼리 자동 감지
- **캐싱 계층 도입**: Redis를 활용한 3단계 캐싱 전략
  - L1: 브라우저 캐시 (정적 데이터)
  - L2: 애플리케이션 캐시 (사용자별 데이터)
  - L3: 데이터베이스 캐시 (공통 조회 데이터)
- **읽기 전용 복제본**: 분석 및 보고서 쿼리를 별도 복제 서버에서 처리
- **수평 확장 준비**: 마이크로서비스 아키텍처로의 점진적 전환 계획

### 리스크 5: 실시간 AI 처리 지연 문제
**상황**: AI 분석 요청 급증으로 인한 응답 지연(목표 500ms → 실제 5초 이상)
**영향도**: 중간 (사용자 경험 저하)
**발생 확률**: 중간
**예상 손실**: 실시간 코칭 기능 사용률 60% 감소

**완화 방안**:
- **비동기 처리 시스템**: 즉시 응답이 필요하지 않은 분석은 백그라운드 처리
- **AI 모델 경량화**: 복잡한 모델을 실시간용(경량)과 배치용(정확)으로 분리
- **큐 시스템 도입**: Redis Queue를 활용한 작업 순서 관리 및 부하 분산
- **예측적 캐싱**: 사용자 패턴 분석을 통한 미리 계산 및 캐싱
- **그레이스풀 폴백**: AI 지연 시 기본 규칙 기반 결과 즉시 제공

## 제품 리스크 (Product Risks)

### 리스크 6: 사용자 온보딩 복잡성으로 인한 이탈
**상황**: AI 기능의 복잡성과 초기 데이터 입력 부담으로 인해 신규 사용자의 70% 이상이 첫 주 내 이탈
**영향도**: 매우 높음 (사용자 확보 실패)
**발생 확률**: 높음 (복잡한 AI 기능의 일반적 문제)
**예상 손실**: 고객 획득 비용 300% 증가, 성장률 80% 감소

**완화 방안**:
- **3단계 점진적 온보딩**:
  - 1단계: 최소 정보만 입력하여 즉시 샘플 AI 리드 경험
  - 2단계: 기존 고객 데이터 간단 임포트 (선택사항)
  - 3단계: 고급 기능 및 맞춤 설정 (1주 후 점진적 안내)
- **즉시 가치 경험 설계**: 회원가입 완료와 동시에 샘플 리드 3-5개 자동 제공
- **인터랙티브 튜토리얼**: 실제 기능을 사용하면서 배우는 실습형 가이드
- **개인화된 온보딩**: 사용자 경험 수준(초보/중급/고급)에 따른 맞춤형 안내
- **1:1 온보딩 지원**: 초기 100명 사용자에게 개인별 온보딩 세션 제공
- **성공 지표 모니터링**: 온보딩 각 단계별 이탈률 실시간 추적 및 개선

### 리스크 7: AI 리드 발굴의 실용성 부족
**상황**: AI가 발굴한 리드의 실제 계약 전환율이 기대치(60%)보다 현저히 낮아(20% 이하) 사용자들이 AI 기능을 불신하게 됨
**영향도**: 매우 높음 (핵심 가치 제안 실패)
**발생 확률**: 중간
**예상 손실**: 핵심 차별화 요소 실패, 일반 CRM으로 격하

**완화 방안**:
- **리드 품질 개선 전략**:
  - 라이프 이벤트 검증 강화: 허위 정보 필터링 알고리즘 개발
  - 접촉 가능성 사전 검증: 연락처 유효성, 접촉 선호도 사전 분석
  - 지역 및 소득 수준 세분화: 타겟 고객층 정교화
- **사용자 기대치 관리**:
  - 명확한 성공률 가이드라인 제시 (초기 20-30%, 6개월 후 50-60%)
  - 전환율보다는 리드 발굴 효율성 강조 (월 5건 → 50건)
  - 성공 사례 지속적 공유 및 베스트 프랙티스 교육
- **지속적 개선 시스템**:
  - 전환 실패 원인 분석 및 모델 개선
  - 성공한 설계사의 접근법 AI 모델에 반영
  - A/B 테스트를 통한 접근 전략 최적화

### 리스크 8: 기존 도구와의 차별화 부족 인식
**상황**: 사용자들이 "기존 CRM + 구글 검색으로도 충분하다"며 고유 가치를 인정하지 않음
**영향도**: 높음 (시장 진입 실패)
**발생 확률**: 중간
**예상 손실**: 시장 침투율 목표치 대비 70% 미달

**완화 방안**:
- **명확한 차별화 메시징**:
  - "하루 5시간 리드 발굴 → 5분 자동 발굴" 구체적 시간 절약 효과 강조
  - 실제 수치 비교: "월평균 15건 리드 → 95건 리드" 정량적 성과 제시
  - 경쟁사 대비 우위 요소 명확화 (AI 예측 vs 단순 검색)
- **실제 성과 증명**:
  - 베타 사용자 성공 사례 수집 및 공개 (수익 증가 %, 시간 절약 시간)
  - ROI 계산기 제공: 도구 비용 대비 수익 증가 효과 시뮬레이션
  - 무료 체험을 통한 직접 경험 기회 제공 (7일 무료 체험)
- **보험업 특화 강조**:
  - 일반 CRM으로는 불가능한 보험업 특수 요구사항 해결
  - 라이프 이벤트 기반 리드 발굴의 독특함 강조
  - 소개 네트워크 시각화의 보험업계 특화 가치 부각

## 비즈니스 리스크 (Business Risks)

### 리스크 9: 초기 사용자 확보 실패
**상황**: 초대장 시스템이 오히려 사용자 확보를 방해하여 목표(6개월 내 500명) 대비 50명만 확보
**영향도**: 매우 높음 (사업 생존 위기)
**발생 확률**: 중간
**예상 손실**: 운영 자금 소진, 사업 중단 위험

**완화 방안**:
- **초기 시드 사용자 전략**:
  - 개인 네트워크 활용: 보험업계 지인 50명 직접 온보딩
  - 보험사 파트너십: AZ보험 등과의 공식 파일럿 프로그램
  - 온라인 커뮤니티 활용: 보험설계사 카페, 단체방 적극 활용
- **초대 인센티브 시스템**:
  - 초대 성공 시 1개월 무료 사용권 제공
  - 활성 사용자 초대 시 추가 초대장 3장 제공
  - 팀 단위 가입 시 팀장에게 특별 혜택 제공
- **유연한 가입 정책**:
  - 베타 기간 중 초대장 없이도 가입 가능
  - 업계 종사자 인증 시 즉시 가입 허용
  - 추천인 시스템으로 품질 관리 병행
- **마케팅 채널 다변화**:
  - 유튜브 인플루언서와 콜라보 (보험업계 유명인)
  - 보험 세미나 및 컨퍼런스 참여
  - 구글/네이버 키워드 광고 (보험설계사 대상)

### 리스크 10: 개인정보 보호 규정 위반
**상황**: AI 리드 발굴 과정에서 개인정보보호법 위반 또는 GDPR 규정 위반으로 인한 법적 제재
**영향도**: 매우 높음 (사업 중단 위험)
**발생 확률**: 낮음 (철저한 준비 시)
**예상 손실**: 과징금 최대 5억원, 사업 중단, 브랜드 이미지 타격

**완화 방안**:
- **법률 자문 및 컴플라이언스**:
  - 개인정보보호 전문 변호사 고문 계약
  - 개인정보보호위원회 사전 상담 및 가이드라인 준수
  - GDPR 컴플라이언스 인증 취득
- **데이터 처리 원칙 수립**:
  - 공개 정보만 사용 원칙 (비공개 개인정보 절대 수집 금지)
  - 데이터 최소화 원칙 (필요 최소한의 정보만 수집)
  - 목적 제한 원칙 (리드 발굴 목적 외 사용 금지)
- **기술적 보안 조치**:
  - 개인정보 암호화 저장 (AES-256)
  - 접근 권한 세분화 및 로그 관리
  - 정기적 보안 감사 및 취약점 점검
- **투명성 및 사용자 권리 보장**:

  - 사용자 데이터 삭제 요청 시 24시간 내 처리
  - 데이터 이용 현황 사용자에게 정기 보고

### 리스크 11: AI 모델의 편향성 및 윤리 문제
**상황**: AI 모델이 특정 지역, 연령, 성별에 대해 편향된 스코어링을 하여 차별 논란 발생
**영향도**: 높음 (브랜드 이미지 및 윤리적 문제)
**발생 확률**: 중간
**예상 손실**: 브랜드 신뢰도 하락, 언론 부정적 보도, 사용자 이탈

**완화 방안**:
- **AI 공정성 설계 원칙**:
  - 편향성 정기 점검 시스템 (월 1회 모델 공정성 분석)
  - 다양한 고객층의 균형 잡힌 학습 데이터 확보
  - 보호 속성(성별, 연령, 지역) 기반 차별 방지 알고리즘
- **투명성 및 설명 가능성**:
  - AI 의사결정 과정 사용자에게 설명 제공
  - 스코어링 요소별 가중치 공개
  - 편향성 의심 시 즉시 신고 가능한 시스템
- **인간 검토 프로세스**:
  - 고득점 리드에 대한 인간 최종 검토
  - 윤리 위원회 구성 (외부 전문가 포함)
  - 정기적 AI 윤리 교육 및 가이드라인 업데이트
- **지속적 모니터링**:
  - 실시간 편향성 지표 추적
  - 사용자 피드백 기반 모델 개선
  - 업계 윤리 기준 적극 준수

### 리스크 12: 수익화 모델의 불확실성
**상황**: 보험설계사들의 디지털 도구 결제 의향이 예상보다 낮아 목표 매출의 30%만 달성
**영향도**: 높음 (지속가능성 위기)
**발생 확률**: 중간
**예상 손실**: 운영비 충당 불가, 투자 유치 실패

**완화 방안**:
- **가치 기반 가격 정책**:
  - 명확한 ROI 제시: "월 19,900원 투자로 평균 50만원 추가 수익"
  - 성과 기반 요금제 도입: 실제 계약 성사 시에만 수수료 부과 옵션
  - 단계별 요금제: 기본 기능 무료, 고급 기능 유료
- **다양한 수익 모델 실험**:
  - B2B 기업 고객 확보 (보험사 직접 계약)
  - 교육 및 컨설팅 서비스 병행
  - 파트너사와의 레베뉴 셰어링 (보험사, 핀테크)
- **사용자 가치 증명**:
  - 무료 체험 기간 중 실제 성과 측정 및 공유
  - 성공 사례 기반 케이스 스터디 제작
  - 동료 추천 프로그램 (입소문 마케팅)

## MVP 정의 및 범위 관리 리스크

### 리스크 13: 범위 크리프 (Scope Creep)
**상황**: 개발 과정에서 "이 기능도 있으면 좋겠다"는 요구사항이 계속 추가되어 출시가 6개월 지연
**영향도**: 높음 (출시 지연으로 인한 기회비용)
**발생 확률**: 높음 (일반적인 개발 과정 문제)
**예상 손실**: 개발비 2배 증가, 시장 기회 상실

**완화 방안**:
- **엄격한 MVP 정의**:
  - 반드시 포함되어야 할 기능만 명시 (AI 리드 발굴 + 기본 고객 관리)
  - "Nice-to-have" vs "Must-have" 명확 구분
  - 기능 추가 시 동등한 기능 제거 원칙
- **정기적 스코프 리뷰**:
  - 주간 개발 회의에서 범위 점검
  - 월간 우선순위 재검토 및 조정
  - 이해관계자 합의 기반 변경 관리
- **시간 박스 적용**:
  - 각 기능에 최대 개발 시간 설정
  - 시간 초과 시 기능 축소 또는 다음 버전 이연
  - 핵심 기능 완성 우선, 부가 기능 후순위

### 리스크 14: 기술 부채 누적
**상황**: 빠른 출시를 위한 임시방편 코드가 누적되어 유지보수 비용이 개발비의 3배로 증가
**영향도**: 중간 (장기적 개발 효율성 저하)
**발생 확률**: 높음 (스타트업 일반적 문제)
**예상 손실**: 개발 속도 50% 저하, 신기능 개발 지연

**완화 방안**:
- **기술 부채 관리 전략**:
  - 스프린트당 20% 시간을 리팩토링에 할당
  - 코드 리뷰 필수화 및 품질 가이드라인 수립
  - 자동화된 테스트 코드 작성 (커버리지 80% 이상)
- **아키텍처 설계 원칙**:
  - 모듈화된 구조로 기능별 독립성 확보
  - 확장 가능한 데이터베이스 스키마 설계
  - API 설계 시 버전 관리 고려
- **지속적 개선 프로세스**:
  - 월간 기술 부채 리뷰 및 우선순위 설정
  - 성능 모니터링을 통한 병목 지점 조기 발견
  - 사용자 피드백 기반 개선 우선순위 결정

#### 데이터 암호화
```sql
-- 민감 정보 암호화 예시
CREATE EXTENSION IF NOT EXISTS pgsodium;

-- 주민등록번호 암호화 함수
CREATE OR REPLACE FUNCTION encrypt_ssn(ssn TEXT)
RETURNS TEXT AS $$
BEGIN
  RETURN pgsodium.crypto_aead_det_encrypt(
    ssn::bytea,
    additional_data := 'ssn'::bytea,
    key_id := 1
  );
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
```

#### RLS 정책 예시
```sql
-- 고객 정보 RLS 정책
ALTER TABLE app_client_profiles ENABLE ROW LEVEL SECURITY;

-- 개인 고객만 조회 가능
CREATE POLICY "Users can view own clients" ON app_client_profiles
  FOR ALL USING (
    user_id = auth.uid() OR 
    team_id IN (
      SELECT team_id FROM app_team_members 
      WHERE user_id = auth.uid()
    )
  );

-- 팀 권한별 차등 접근
CREATE POLICY "Team admins can manage team clients" ON app_client_profiles
  FOR ALL USING (
    team_id IN (
      SELECT team_id FROM app_team_members 
      WHERE user_id = auth.uid() AND role IN ('team_admin', 'senior')
    )
  );
```
